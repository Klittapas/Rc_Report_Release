ตอนที่ 1

โจทย์: วันไหนเป็นวันแบบไหน

ทีม Revenue ดูข้อมูลรายวันของโรงแรม 7 แห่ง แล้วอยากรู้ว่า "วันแบบไหนที่พฤติกรรมเหมือนกัน" — วันพีคที่ห้องเต็ม ADR สูง, วันธรรมดาที่ขายเรื่อยๆ, วันเงียบที่ต้องอัดโปร ถ้าจัดกลุ่มได้ ก็ตั้งราคาเป็นชุดตามกลุ่มได้ ไม่ต้องนั่งไล่ทีละวัน

ปัญหาคือ ไม่มีใครมีเฉลย ว่าวันที่ 14 ก.ค. เป็น "วันพีค" หรือ "วันธรรมดา" — ไม่มีคอลัมน์ label ในไฟล์ CSV งานแบบนี้เรียกว่า

Unsupervised learning — ให้เครื่องหากลุ่มเอง จากความเหมือนของข้อมูลล้วนๆ ไม่มีเฉลยให้เทียบ ตรงข้ามกับ supervised (เช่น ทายราคาบ้าน) ที่มีคำตอบจริงไว้สอน

k-means คืออัลกอริทึม unsupervised ที่เก่าแก่และเร็วที่สุดตัวหนึ่ง หน้าที่เดียวของมันคือ: รับจุด n จุด → แบ่งเป็น k กอง โดยให้จุดในกองเดียวกันอยู่ใกล้กันที่สุด

ตอนที่ 2

ก่อนจะจัดกลุ่มอะไรได้ ต้องแปลงของจริง (วัน) ให้เป็นตัวเลขก่อน ใน DayTypeClusters.tsx เรายุบข้อมูลทั้งวันเหลือตัวเลข 6 ตัวต่อวัน:

ADRราคาเฉลี่ยต่อห้อง (rev ÷ rooms)฿1,484

%NRFสัดส่วน non-refundable0.41

พอทำแบบนี้ แต่ละวันกลายเป็นจุด 1 จุดในอวกาศ 6 มิติ วาดไม่ได้ด้วยตา แต่คณิตศาสตร์ไม่สนใจ — สูตรเดียวกับ 2 มิติทุกอย่าง แค่บวกเพิ่มอีก 4 ช่อง

วิธีคิดสำคัญ: การเลือก feature คือการเลือกว่า "ความเหมือน" แปลว่าอะไร เราใส่ %NRF เข้าไป = เราประกาศว่าวันที่ลูกค้ากล้าจองแบบคืนเงินไม่ได้ ควรถูกจัดเป็นพวกเดียวกัน ถ้าเปลี่ยนชุด feature กลุ่มจะเปลี่ยนตาม — ก่อนโทษอัลกอริทึม ให้ดู feature ก่อนเสมอ

ตอนที่ 3

ความเหมือน = ระยะทาง

"วันสองวันเหมือนกันแค่ไหน" ตอบด้วย ระยะทางแบบยุคลิด (Euclidean) — สูตรพีทาโกรัสที่ยืดไปกี่มิติก็ได้:

// cluster.ts:31 — ระยะทางยกกำลังสอง (ไม่ถอดรูทเพื่อความเร็ว การเปรียบเทียบให้ผลเท่ากัน)
const dist2 = (a, b) => {
  let s = 0;
  for (let j = 0; j < a.length; j++) s += (a[j] - b[j]) ** 2;
  return s;
};

จุดที่ระยะน้อย = วันที่พฤติกรรมคล้ายกัน นี่คือนิยาม "เหมือน" ทั้งหมดที่ k-means รู้จัก มันไม่เข้าใจโรงแรม ไม่เข้าใจ ADR — เข้าใจแค่ระยะห่างระหว่างตัวเลข

ตอนที่ 4

กับดักหน่วย → ทำไมต้อง z-score

ลองวัดระยะดิบๆ ระหว่างสองวันจริงจากข้อมูลเดือนกรกฎาคมของเรา:

roomsADR%OTA
อังคาร 7 ก.ค.1,6601,4960.85
อาทิตย์ 5 ก.ค.9891,4840.72
ผลต่าง²450,2411440.0169

เห็นปัญหาไหม — ผลต่างของ rooms ใหญ่กว่าผลต่างของ %OTA ถึง 26 ล้านเท่า ถ้าปล่อยแบบนี้ ระยะทางทั้งก้อนจะถูก rooms ตัวเดียวกลบหมด %OTA, %NRF, weekend จะเหมือนไม่มีตัวตน — จัดกลุ่มด้วย rooms ล้วนๆ โดยไม่รู้ตัว

ทางแก้มาตรฐานคือ z-score: แปลงทุก feature ให้อยู่สเกลเดียวกัน — "ห่างจากค่าเฉลี่ยกี่เท่าของส่วนเบี่ยงเบนมาตรฐาน (SD)"

// cluster.ts:20 — standardize: (ค่า − mean) ÷ SD ทีละคอลัมน์
export function standardize(data: number[][]) {
  // ... หา means[j], sds[j] ของแต่ละคอลัมน์
  const z = data.map((row) => row.map((v, j) => (v - means[j]) / sds[j]));
  return { z, means, sds };
}

หลังแปลง ทุก feature พูดภาษาเดียวกัน: +2.0 แปลว่า "สูงผิดปกติ 2 SD" ไม่ว่าจะเป็นห้องหลักพันหรือสัดส่วน 0–1 ตอนนี้ %OTA มีสิทธิ์มีเสียงเท่า rooms แล้ว

ถ้าลืมขั้นนี้: k-means ยังรันได้ ไม่มี error อะไรเลย — แต่กลุ่มที่ได้จะเป็นแค่ "เรียงตามจำนวนห้อง" ในคราบ clustering นี่คือบั๊กเงียบที่พบบ่อยที่สุดของ k-means

ตอนที่ 5

หัวใจ k-means: วนสองจังหวะ

อัลกอริทึมทั้งหมดคือลูปเดียว สลับสองจังหวะจนกว่าจะนิ่ง:

จังหวะ 1 — Assign: แต่ละจุดวิ่งเข้าหา centroid (จุดศูนย์กลางกลุ่ม) ที่ใกล้ตัวเองที่สุด

จังหวะ 2 — Update: centroid แต่ละตัวย้ายไปอยู่ตรงค่าเฉลี่ยของสมาชิกตัวเอง

วนซ้ำ จนไม่มีจุดไหนย้ายกลุ่ม = ลู่เข้า (converged)

// cluster.ts:64 — จังหวะ assign
for (let i = 0; i < n; i++) {
  let bi = 0, bd = Infinity;
  for (let c = 0; c < kk; c++) {
    const dd = dist2(z[i], centroids[c]);
    if (dd < bd) { bd = dd; bi = c; }          // หา centroid ที่ใกล้สุด
  }
  if (assign[i] !== bi) { assign[i] = bi; moved = true; }
}
// cluster.ts:74 — จังหวะ update: centroid = ค่าเฉลี่ยของสมาชิก
centroids = sums.map((s, c) => counts[c] ? s.map((v) => v / counts[c]) : centroids[c]);
if (!moved && it > 0) break;                    // ไม่มีใครย้าย = จบ

ลองเล่นเอง — ดู centroid เดินทีละก้าว

กด "วาง centroid" เพื่อเริ่ม

จุด = วัน (จำลอง 2 มิติให้เห็นด้วยตา) · ✕ = centroid · สี = กลุ่มปัจจุบัน · เส้นจาง = ระยะจากจุดถึง centroid ของมัน · inertia = ผลรวมระยะ² ยิ่งต่ำยิ่งกระชับ

สังเกตตอนกด "เดิน 1 รอบ" ซ้ำๆ: centroid จะขยับน้อยลงเรื่อยๆ แล้วหยุดสนิท — ทุกครั้งที่วน inertia มีแต่ลดลง (พิสูจน์ได้ทางคณิตศาสตร์) มันเลยการันตีว่าหยุดแน่ แต่…ไม่การันตีว่าหยุดในจุดที่ดีที่สุด ซึ่งนำเราไปตอนถัดไป

ตอนที่ 6

จุดเริ่มชี้ชะตา → k-means++ และ 12 restarts

ลองกด "วาง centroid (สุ่ม)" ในเดโมข้างบนหลายๆ ครั้งแล้วรันจนจบ — บางครั้งได้กลุ่มสวย บางครั้ง centroid สองตัวไปเกิดติดกันแล้วแย่งกองเดียวกัน ส่วนอีกกองโดนควบรวม ผลสุดท้ายขึ้นกับดวงตอนวางจุดเริ่ม ปรากฏการณ์นี้เรียกว่าติด local minimum

โค้ดเราแก้สองชั้น:

ชั้นที่ 1 — k-means++ : สุ่มแบบมีชั้นเชิง

// cluster.ts:40 — seedPP: จุดที่ "ไกลจาก centroid ที่มีแล้ว" มีโอกาสถูกเลือกสูง
const d2 = z.map((p) => Math.min(...centroids.map((c) => dist2(p, c))));
let r = rand() * total;   // สุ่มถ่วงน้ำหนักด้วยระยะ² → centroid เริ่มต้นกระจายตัว

แทนที่จะสุ่มมั่ว มันเลือก centroid ตัวถัดไปแบบ "ยิ่งไกลจากตัวที่มีอยู่ ยิ่งน่าเลือก" — โอกาสที่สองตัวเกิดติดกันจึงต่ำมาก

ชั้นที่ 2 — วิ่ง 12 รอบ เก็บรอบที่ดีที่สุด

// cluster.ts:54 — restarts = 12: รันทั้งกระบวน 12 ครั้ง คนละจุดเริ่ม
// cluster.ts:78 — เก็บครั้งที่ inertia ต่ำสุดเท่านั้น
if (!best || inertia < best.inertia) best = { assignments, centroids, inertia };

และมีรายละเอียดเล็กๆ ที่คิดมาดี: seed = 42 ตายตัว — เลขสุ่มเลยออกลำดับเดิมทุกครั้ง ผลจัดกลุ่มจึงเหมือนเดิมทุกการ re-render ไม่งั้นสี cluster บน dashboard จะกระพริบสลับกลุ่มทุกครั้งที่ React วาดใหม่

ตอนที่ 7

เลือก k ยังไง — elbow กับ silhouette

k-means มีจุดอ่อนโครงสร้าง: เราต้องบอกจำนวนกลุ่ม (k) ให้มันก่อน และมันจะแบ่งให้ครบ k กองเสมอ ต่อให้ข้อมูลไม่มีกลุ่มจริงก็ตาม ใน dashboard เราให้ผู้ใช้กดเลือก k = 2 / 3 / 4 เอง แล้วจะรู้ได้ไงว่าอันไหน "ถูก"?

เครื่องมือ 1 — Elbow (ดู inertia)

เพิ่ม k ไปเรื่อยๆ inertia ลดลงเสมอ (กองเยอะขึ้น แต่ละกองย่อมกระชับขึ้น) จุดที่กราฟ "หักศอก" — ลดต่อแล้วได้กำไรน้อย — คือ k ที่เหมาะ

เครื่องมือ 2 — Silhouette (−1 ถึง 1)

วัดต่อจุด: ฉันใกล้เพื่อนร่วมกลุ่ม (a) มากกว่าใกล้กลุ่มข้างบ้าน (b) แค่ไหน = (b−a)/max(a,b) แล้วเฉลี่ยทั้งชุด

silhouetteอ่านว่า
> 0.50กลุ่มชัด แยกขาดจริง
0.35 – 0.50พอใช้ — มีโครงสร้างแต่ขอบเบลอ
0.25 – 0.35อ่อน — เริ่มคาบเกี่ยว
< 0.25แทบไม่มีกลุ่มจริง k-means แค่หั่นตามหน้าที่

กราฟตัวอย่างที่เราวาดกัน (ข้อมูลจำลอง 90 วัน, k=3) ได้ silhouette 0.46 — ระดับ "พอใช้–ดี": สามวงแยกจากกันเห็นชัด แต่ในวงยังหลวม

ช่องโหว่ใน dashboard ตอนนี้: เราให้ผู้ใช้เลือก k เองแต่ไม่โชว์ silhouette — ผู้ใช้ไม่มีทางรู้เลยว่ากลุ่มที่เห็นคือโครงสร้างจริงหรือการหั่นตามคำสั่ง นี่คืออัปเกรดที่ควรทำที่สุดถ้าจะใช้งานจริง

ตอนที่ 8

อ่านผลจริงของเรา: ทำไมออกมาเป็นแบบนี้

รันบนข้อมูลของเราแล้วมักได้ 3 กลุ่มหน้าตาประมาณนี้ (เรียงตาม rooms เฉลี่ย แล้วตั้งชื่อในโค้ด):

กลุ่มrooms/วันADR%NRFนิสัย
Peak demandสูงสูงสูงเสาร์–อาทิตย์ / วันหยุดยาว
High demandกลางกลางกลางวันธรรมดาช่วงคึกคัก
Steadyต่ำต่ำต่ำวันเงียบ ต้องพึ่งโปร

ทำไมถึงออกทรงนี้เสมอ? ตอนเราวาด PCA (ยุบ 6 มิติเหลือ 2 เพื่อดูด้วยตา) แกนแรกกินความแปรปรวนไป 78% — แปลว่า feature ทั้ง 6 ของเรา แทบทั้งหมดเคลื่อนไปทางเดียวกัน: วันที่ห้องเยอะ → ADR ก็สูง → %NRF ก็สูง → มักเป็น weekend มิติที่ดูเหมือนมี 6 จริงๆ เหลือมิติเด่นอยู่มิติเดียว คือ "ระดับดีมานด์"

บทเรียนสำคัญที่สุดของทั้งเรื่อง: k-means ไม่ได้ "ค้นพบ" อะไรลึกลับ — มันสะท้อนโครงสร้างของ feature ที่เราป้อน เมื่อ feature ทุกตัวสัมพันธ์กันสูง ผลจัดกลุ่มจะเป็นการหั่นสเกลเดียว (พีค/กลาง/เงียบ) ซึ่ง…เรียงตาม rooms เฉยๆ ก็ได้ผลใกล้เคียงกัน ค่าของ k-means ที่นี่คือมันหาเส้นแบ่งเองอย่างเป็นระบบ ไม่ใช่เราขีดเส้นตามใจ

ตอนที่ 9

ข้อจำกัดที่ต้องรู้ ก่อนเชื่อกราฟ

1 · weekend เป็นเลข 0/1 — พอ z-score แล้วตัวแปรสองขั้วนี้แรงมาก มักลากกลุ่มให้กลายเป็น "แยกเสาร์อาทิตย์ vs วันธรรมดา" กลบ signal อื่น ทางแก้: ตัดออกจาก feature แล้วใช้เป็น filter แทน หรือถ่วงน้ำหนักให้เบาลง

2 · feature ซ้ำทางกัน — rooms / ADR / %NRF ไปด้วยกันหมด (ตอนที่ 8) กลุ่มเลยได้แค่แกนดีมานด์ ทางแก้: เพิ่ม feature ที่ตั้งฉากกัน เช่น lead time เฉลี่ย, สัดส่วน cancel

3 · วันน้อยเกิน = ผลมั่ว — ช่วงกรอง 16 วัน หาร k=4 เหลือกลุ่มละ ~4 วัน สถิติอะไรก็ไม่นิ่ง ทางแก้: บังคับขั้นต่ำ ~60 วันก่อนยอมรัน (ตอนนี้โค้ดเช็คแค่ days ≥ k)

4 · ไม่มีตัวเลขความเชื่อมั่นบนจอ — ควรคำนวณ silhouette แล้วโชว์คู่ผลเสมอ ให้ผู้ใช้แยก "กลุ่มจริง" ออกจาก "หั่นตามคำสั่ง" ได้

แปลงวันเป็นเวกเตอร์ 6 ช่อง → z-score ให้ทุก feature เสียงเท่ากัน → k-means วนสองจังหวะ (assign/update) จน inertia นิ่ง → กันดวงซวยด้วย k-means++ บวก 12 restarts และ seed ตายตัว → เลือก k ด้วย elbow/silhouette ไม่ใช่ความรู้สึก → แล้วอ่านผลอย่างมีวิจารณญาณ: ถ้า feature สัมพันธ์กันหมด กลุ่มที่ได้ก็คือสเกลเดียวถูกหั่นเป็นท่อน ไม่ใช่การค้นพบมหัศจรรย์ — เข้าใจครบเท่านี้ ก็อ่าน cluster.ts ทั้งไฟล์ออก และรู้ว่าจะพัฒนามันต่อตรงไหน
